Database Configuration

Database Overview

The Trellix platform requires a properly configured Microsoft SQL Server database for optimal performance and data storage. This guide covers setting up SQL Server for Trellix, configuring replication, and optimizing performance for enterprise deployments.

Note: This guide assumes you have administrative access to your SQL Server environment and the necessary permissions to create databases and configure server settings.

Microsoft SQL Server for Trellix Official

Recommended: SQL Server 2019 Critical for Trellix Platform
SQL Server Documentation

Pre-requisites

  • Microsoft SQL Server 2019+ - Standard or Enterprise edition
  • Minimum 16GB RAM allocated to database services
  • 100GB+ storage with high I/O performance (SSD recommended)
  • Network connectivity between Trellix server and database
  • SQL Server Authentication - Mixed Mode authentication enabled

Standard SQL Server Setup

1

Install Microsoft SQL Server

Install SQL Server with Mixed Mode authentication enabled. Ensure TCP/IP protocol is enabled in SQL Server Configuration Manager.

2

Create Trellix Database

Set the database collation to SQL_Latin1_General_CP1_CI_AS for compatibility with Trellix components.

CREATE DATABASE TrellixEPO COLLATE SQL_Latin1_General_CP1_CI_AS;
3

Configure Database User

Create a dedicated database user with appropriate permissions for the Trellix application.

CREATE LOGIN TrellixUser WITH PASSWORD = 'StrongPassword123!'; USE TrellixEPO; CREATE USER TrellixUser FOR LOGIN TrellixUser;
4

Configure SQL Server Properties

Set SQL Server memory limits, recovery model, and auto-growth settings based on your environment scale.

-- Set memory limits EXEC sp_configure 'show advanced options', 1; RECONFIGURE; EXEC sp_configure 'max server memory (MB)', 12288; RECONFIGURE;
5

Configure Maintenance Plans

Set up SQL Server maintenance plans for index maintenance, statistics updates, and backups.

Performance Optimization

For optimal Microsoft SQL Server performance with Trellix products:

Properly size data and log files

Pre-allocate files to their expected size to reduce auto-growth events

Use multiple TempDB files

Configure one TempDB file per CPU core (up to 8) to reduce contention

Place files on separate volumes

Put data files, log files, and TempDB on separate physical storage

Ensure proper indexing

Monitor query performance and add indexes to frequently queried columns

Regular maintenance

Follow the maintenance plan outlined above

Best Practice for Large Deployments

For large Trellix deployments (10,000+ endpoints), consider implementing SQL Server Resource Governor to protect critical database resources and ensure consistent performance during peak load periods.

Data Exchange Layer (DXL)

Advanced
30 minutes

What is DXL?

The Data Exchange Layer (DXL) is a messaging framework that enables real-time communication and integration between security components in the Trellix ecosystem. It allows for seamless data sharing and automated responses across your security infrastructure.

Note: DXL is a critical component for enabling communication between various Trellix products and third-party security solutions.

Trellix Data Exchange Layer Official

Real-time Communication Security Integration
Documentation

Key Benefits

  • Real-time information sharing between security products
  • Automated security responses to threats
  • Integration with third-party security tools
  • Reduction in manual security operations
  • Scalable message bus architecture

DXL Architecture

DXL uses a publish-subscribe messaging pattern that allows components to publish events and subscribe to topics. This enables efficient, real-time communication across your security infrastructure.

Core Components

  • DXL Brokers: Message routing servers that handle communication
  • DXL Clients: Security products and tools that connect to the DXL fabric
  • DXL Topics: Categorized channels for message distribution
  • DXL Services: Request/response capabilities for direct interactions

Setting Up DXL

1

Install DXL Broker

Install the DXL Broker on a dedicated server or virtual machine. Ensure it has network connectivity to all security components.

# Example installation on Linux sudo rpm -ivh dxlbroker-version.rpm # Initialize the broker sudo /opt/dxlbroker/bin/dxlbroker config
2

Configure Certificates

Set up certificate-based authentication for secure DXL communications.

# Generate certificates sudo /opt/dxlbroker/bin/dxlbroker cert generate # Export client certificates sudo /opt/dxlbroker/bin/dxlbroker cert clientcert client1
3

Start DXL Broker Service

Start the DXL broker service and verify it's running correctly.

# Start the broker service sudo systemctl start dxlbroker # Verify status sudo systemctl status dxlbroker
4

Configure DXL Clients

Configure Trellix products and third-party tools to connect to the DXL fabric.

# Example Python client configuration from dxlclient.client import DxlClient from dxlclient.client_config import DxlClientConfig # Create configuration from file config = DxlClientConfig.create_dxl_config_from_file("dxlclient.config") # Create the client client = DxlClient(config) client.connect()
5

Verify Communication

Test the DXL fabric to ensure messages are being properly routed between components.

# Example Python test script from dxlclient.callbacks import EventCallback # Define callback for events class MyEventCallback(EventCallback): def on_event(self, event): print("Received event: " + event.payload.decode()) # Register callback for a topic client.add_event_callback("my/test/topic", MyEventCallback())

DXL Integration Examples

Threat Intelligence Sharing

Configure TIE to share file reputation data over DXL with other security components:

# Subscribe to TIE file reputation events client.add_event_callback( "/trellix/service/tie/file/reputation", FileReputationCallback() )

Automated Response

Configure automated actions when threats are detected:

# When malware detected, isolate endpoint def on_malware_event(event): system_guid = extract_guid(event) isolate_endpoint(system_guid) client.add_event_callback( "/trellix/event/ens/threat/malware", MalwareEventCallback(on_malware_event) )

Best Practices

  • High Availability: Deploy multiple DXL brokers for redundancy
  • Security: Use TLS and certificate authentication for all communications
  • Performance: Monitor broker performance and scale as needed
  • Topic Design: Create a logical topic hierarchy for efficient message routing
  • Error Handling: Implement robust error handling in all DXL clients

Important Consideration

DXL brokers are critical infrastructure components. Ensure they are properly secured, monitored, and included in business continuity plans.

Data Loss Prevention

Advanced
45 minutes

What is Data Loss Prevention?

Trellix Data Loss Prevention (DLP) is a comprehensive solution designed to discover, monitor, and protect sensitive data across your organization. It helps prevent data exfiltration, insider threats, and accidental exposure of confidential information by enforcing content-aware security policies.

Note: DLP is crucial for compliance with regulations such as GDPR, HIPAA, PCI DSS, and other data protection requirements, helping organizations avoid costly data breaches and regulatory penalties.

Data Loss Prevention Official

Data Protection Compliance
Documentation

Key Features

  • Data Discovery: Locate and classify sensitive data across endpoints and servers
  • Real-time Protection: Monitor and block unauthorized data transfers
  • Content Inspection: Analyze data content, not just file types
  • Policy Management: Create and enforce granular data protection policies
  • Incident Management: Track and remediate data policy violations
  • User Education: Provide feedback to users about policy violations

Architecture Components

The Trellix DLP solution consists of multiple components working together to provide comprehensive data protection:

Core Components

  • DLP Endpoint: Agent software that monitors data usage on endpoints
  • DLP Server: Central management server that enforces policies
  • ePO Integration: Management console for configuration and reporting
  • Policy Manager: Interface for creating and managing DLP rules
  • Incident Manager: Tool for reviewing and responding to violations

Deploying DLP

1

Prepare Infrastructure

Ensure your environment meets the necessary requirements:

# System Requirements - Trellix ePO 5.10 or higher - SQL Server 2016 or higher - Minimum 16GB RAM for DLP Server - 4-core CPU or higher - 500GB+ storage for policy data and incidents - Windows Server 2016/2019/2022
2

Install DLP Extensions

Add DLP components to your ePO server:

1. In ePO, navigate to Menu > Software > Extensions 2. Click "Check for New Extensions" 3. Locate and select "Data Loss Prevention" 4. Click "Install" and follow the installation wizard 5. Restart ePO services when prompted
3

Configure DLP Server

Set up the DLP server with appropriate settings:

1. Navigate to Menu > Data Protection > Settings 2. Configure general settings: - Performance optimization - Storage locations - Notification templates - User feedback options 3. Set up classification definitions 4. Configure incident handling rules 5. Apply and save settings
4

Create Classification Definitions

Define types of sensitive data to protect:

1. Navigate to Menu > Data Protection > Classifications 2. Create common classifications: - PII (Personally Identifiable Information) - PHI (Protected Health Information) - PCI (Payment Card Information) - Intellectual Property - Confidential Documents 3. Configure detection methods: - Regular expressions - Dictionary matching - Document properties - Custom scripts 4. Test classifications with sample data 5. Save and apply classifications
5

Create and Deploy Policies

Create policies to protect classified data:

1. Navigate to Menu > Policy > Policy Catalog 2. Select "Data Loss Prevention" product 3. Create a new policy set 4. Add rules for different channels: - Email protection - Web/cloud uploads - Removable media - Print/screen capture - Application control 5. Define actions for violations: - Block action - Warn user - Encrypt data - Log only 6. Assign to appropriate systems 7. Enable policy and deploy

Implementing Common DLP Policies

Email Protection Policy

Create a policy to prevent sensitive data from being sent via email:

Policy Example: 1. Rule Name: "Prevent PII in External Emails" 2. Classifications: PII, SSN, Credit Card Numbers 3. Conditions: - Channel: Email - Direction: Outbound - Recipients: External domains not in whitelist 4. Actions: - Block email if more than 5 PII items detected - Warn user if 1-5 PII items detected - Log all incidents - Notify security team for high-severity violations

Removable Media Policy

Create a policy to control data transfers to USB devices:

Policy Example: 1. Rule Name: "Secure Removable Media Transfers" 2. Classifications: All sensitive data types 3. Conditions: - Channel: Removable Media - Device Types: USB Drives, External HDDs 4. Actions: - Block unauthorized devices - Encrypt sensitive files when copied to approved devices - Log all file transfers - Require justification for transfers over 10MB

Managing DLP Incidents

Once policies are deployed, you'll need to manage incidents and violations:

Incident Workflow

  1. Detection: Policy violation occurs and is recorded
  2. Notification: Alerts sent to configured recipients
  3. Investigation: Security team reviews incident details
  4. Classification: Incident categorized (false positive, severity level, etc.)
  5. Remediation: Actions taken to address the violation
  6. Documentation: Notes and resolution details added
  7. Closure: Incident marked resolved after remediation

False Positive Management

DLP solutions often generate false positives, especially during initial deployment. Plan to allocate resources for tuning policies and managing incidents during the first few weeks of implementation to maintain productivity while improving security.

Best Practices

  • Phased Deployment: Start with monitoring mode before enforcing blocking actions
  • User Education: Train employees on data handling policies and DLP notifications
  • Policy Tuning: Regularly review and adjust policies to reduce false positives
  • Business Exceptions: Create exception processes for legitimate business needs
  • Regular Auditing: Review incident patterns and policy effectiveness
  • Data Classification: Implement a formal data classification program

Understanding Client Tasks

Client Tasks in Trellix ePO allow administrators to schedule and execute various operations on managed endpoints. These tasks include product deployment, configuration updates, security scans, and maintenance activities across your organization.

Note: Client Tasks are a critical component for managing endpoint security and ensuring consistent protection across your environment.

Trellix ePO Client Tasks Official

Endpoint Management Scheduled Operations
Documentation

Types of Client Tasks

ePO offers several types of client tasks based on your security needs:

  • Product Deployment: Install or update security products
  • On-Demand Scan: Initiate immediate security scans
  • Product Configuration: Update endpoint security settings
  • Agent Wake Up: Force agent check-in with ePO server
  • Collection: Gather information from endpoints
  • Custom Tasks: Execute specialized operations

Creating Client Tasks

1

Access the Client Task Catalog

Navigate to the Client Task Catalog in the ePO console:

Menu > Automation > Client Task Catalog
2

Create a New Client Task

Select the type of task you want to create and define its parameters:

1. Click "New Client Task" 2. Select product from the dropdown (e.g., Endpoint Security) 3. Select task type (e.g., On-Demand Scan) 4. Name the task (e.g., "Weekly Full Scan") 5. Configure task-specific settings
3

Configure Task Settings

Define the specific parameters for your selected task:

# Example: On-Demand Scan settings Scan Type: Full Scan Scan Locations: All Local Drives Schedule: Weekly, Sunday at 1:00 AM Maximum Runtime: 4 hours Action on Detection: Clean, Quarantine if cannot clean
4

Assign the Task

Specify which systems should receive the task:

1. Go to System Tree or Query Results 2. Select target systems or groups 3. Right-click and choose "Client Task" 4. Select your newly created task 5. Set schedule parameters (run now or schedule)
5

Monitor Task Execution

Track the status and results of your client tasks:

Menu > Automation > Client Tasks # Filter options - Status (Running, Completed, Failed) - Task Type - Time Range - Target Systems

Common Client Task Examples

Weekly Security Scan

Schedule a comprehensive security scan during non-business hours:

Task Type: On-Demand Scan (ENS) Name: Weekly Full Scan Settings: - Scan Type: Full Scan - All local drives - Scan memory and startup items Schedule: Every Sunday at 2:00 AM Target: All Workstations

Endpoint Software Deployment

Deploy or update security software across the organization:

Task Type: Product Deployment Name: ENS 10.7.0 Upgrade Package: ENS_10.7.0_Patch3.zip Settings: - Force reboot: No - Retry Count: 3 - Installation timeout: 30 minutes Target: Staged deployment groups

Agent Wake-Up

Force ePO Agents to check in immediately:

Task Type: Agent Wake-Up Name: Immediate Check-In Settings: - Priority: High Target: Systems with outdated policy

Best Practices

  • Staggered Scheduling: Avoid scheduling resource-intensive tasks for all systems simultaneously
  • Testing: Test client tasks on a small group before organization-wide deployment
  • Monitoring: Regularly review task status and completion rates
  • Documentation: Maintain documentation of all scheduled tasks and their purposes
  • Cleanup: Remove or disable obsolete tasks to avoid confusion

Important Consideration

Resource-intensive tasks like full system scans can impact system performance. Schedule these tasks during periods of minimal user activity.

Best Practices

  • Incident Response: Incorporate TIE into incident response procedures
  • Monitoring: Regularly check TIE dashboards for system health and activity

What are Server Tasks?

Server Tasks in Trellix ePO are scheduled operations that automate critical functions across your security infrastructure. They enable administrators to automatically deploy agents, update product components, synchronize directory data, run reports, and perform system maintenance on a defined schedule, ensuring your security environment remains current and operational.

Note: Effective server task management is crucial for maintaining a healthy Trellix environment. Properly configured tasks reduce administrative overhead while ensuring security components are consistently updated and operational.

  • Task Scheduling: Configure one-time or recurring operations
  • Agent Deployment: Push agent installations to new systems
  • Product Updates: Distribute product extensions and updates
  • Directory Synchronization: Sync with Active Directory and LDAP
  • Report Generation: Automatically run and distribute reports
  • Database Maintenance: Purge outdated data and optimize performance

Types of Server Tasks

Trellix ePO offers several categories of server tasks to support different operational needs:

Database Maintenance Tasks

Regular database maintenance is essential for optimal ePO performance:

Critical Maintenance Tasks

  • Purge Events: Remove outdated event data to control database size
  • Database Backup: Create regular backups of the ePO database
  • Index Optimization: Rebuild indexes for improved query performance
  • Log Cleanup: Remove old log files to free disk space
  • Consistency Checks: Verify database integrity

Database Size Management

Without proper purge tasks, the ePO database can grow extremely large, leading to performance degradation and backup failures. Implement appropriate event retention policies based on your compliance requirements and storage capabilities. Most organizations should retain detailed event data for 30-90 days, with summary data kept for longer periods.

  • Notification Setup: Configure alerts for critical task failures
  • Testing: Test new tasks on a limited system group before full deployment
  • Documentation: Maintain records of all scheduled tasks and their purposes
